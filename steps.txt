Deployment (uploading the function code) is separate from execution — deploying doesn't run it; it only becomes live once the pg_cron job or a manual call triggers it.

Plan
1. Local stack setup
npx supabase init (scaffolds config.toml; existing supabase/functions/ is picked up automatically).
npx supabase start — brings up local Postgres/Auth/Edge Runtime in Docker. Note the printed local API URL, anon key, and DB connection string from npx supabase status.
Apply the schema to the local DB: run supa.sql (habits, checkins, user_stats, goal_wallets, credit_habit_wallet function, last_credited_date column — the pg_cron block at the bottom will simply no-op/error harmlessly locally since pg_cron/pg_net/Vault aren't relevant locally; that block can be skipped when applying locally, or left — it's idempotent-safe to skip for this test) against the local Postgres instance via psql.
2. Seed fixture data
Create supabase/seed.sql (also useful going forward for local dev, not just this test) with:

A test row in auth.users (minimal columns: id, aud, role, email, encrypted_password, instance_id='00000000-0000-0000-0000-000000000000', timestamps) — required because habits/checkins/goal_wallets all FK to auth.users(id).
One active habit (daily_units = 5, cost_per_unit = 15 → expected credit 75).
One clean check-in dated inside the IST day the function will treat as "yesterday" at test time — computed relatively in SQL (e.g. ((now() at time zone 'Asia/Kolkata')::date - 1 + time '12:00') at time zone 'Asia/Kolkata') rather than a hardcoded date, so the seed stays correct regardless of when the test actually runs.
One goal_wallets row for that habit (current_balance = 0, is_locked = true, last_credited_date = null).
3. Manual invocation — verify balance increments
npx supabase functions serve daily-wallet-credit (local CLI auto-injects local SUPABASE_URL/SUPABASE_SERVICE_ROLE_KEY into the function — no manual secret wiring needed).
curl -i --request POST 'http://127.0.0.1:54321/functions/v1/daily-wallet-credit' -H "Authorization: Bearer <local anon key>".
Inspect the JSON response (credited, skipped, errors) and query goal_wallets before/after via psql — confirm current_balance increased by exactly 75 and last_credited_date is set to the expected date.
4. Idempotency check
Call the same curl command again immediately.
Confirm the JSON response now shows credited: 0, skipped: 1, and current_balance in the DB is unchanged from step 3 — proves credit_habit_wallet's guarded UPDATE correctly no-ops on a repeat call for the same day.
5. Tear down local stack
npx supabase stop once both checks pass.
6. Deploy to the real project
Only after local tests pass, and only if CLI auth is available (see blocker above):

npx supabase link --project-ref ymoalikzfqavfcyczpvp
npx supabase functions deploy daily-wallet-credit
Explicitly out of scope for this task: invoking the deployed function against the real project's data.
